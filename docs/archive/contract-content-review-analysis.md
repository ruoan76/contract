### 1.2 缺失的核心设计

| 缺失项 | 严重度 | 影响 |
|--------|--------|------|
| **条款 ↔ 审查条目映射机制** | 🔴 致命 | `clause_id` 如何生成、如何保持跨版本稳定、如何锚定到原文位置，均未设计 |
| **合同内容展示布局** | 🔴 致命 | 评审人看合同原文 + AI 标记 + 自己批注，三者未在一屏联动 |
| **起草人修订工作台** | 🔴 致命 | 退回修改后，起草人如何看建议、如何逐条修改、如何重新提交，均无设计 |
| **行内批注能力** | 🟠 高 | 类似 Google Docs 的“建议模式”（Suggesting），评审人直接在原文上标记修改 |
| **修改建议接受/拒绝流** | 🟠 高 | 起草人逐条 Accept/Reject 修改建议，生成新版合同，而非全文重写 |
| **审查发现 ↔ 修改结果闭环** | 🟠 高 | AI 发现的每条风险 → 人工意见 → 起草人是否修改 → 是否消除风险，链路不通 |

**核心交互规则：**

| 交互 | 行为 |
|------|------|
| **点击左侧条款** | 右侧自动滚动到该条款的审查面板 |
| **在原文上划选文字** | 弹出"添加批注/标记风险"工具栏 |
| **点击 AI 标记的风险点** | 左侧原文对应条款高亮（黄色/红色背景） |
| **点击"确认"** | 标记为"法务已确认"，颜色从红→绿 |
| **点击"修正"** | 弹出编辑框，修改 AI 的风险等级/描述/建议 |
| **点击"忽略"** | 标记为"无风险"，颜色从红→灰 |
| **底部结论联动** | 所有 AI 标记处理完毕后，"有条件通过"按钮才可点击 |

### 2.2 条款 ID 体系设计

```python
class ClauseIdentifier:
    """
    条款唯一标识：跨版本稳定，支持位置锚定
    
    格式：{contract_id}:{section_number}:{paragraph_index}
    示例：1001:4:1（合同1001，第四条，第1款）
    """
    
    @staticmethod
    def generate(contract_id: int, section_order: int, para_index: int) -> str:
        return f"{contract_id}:{section_order}:{para_index}"

    @staticmethod
    def parse(clause_id: str) -> Tuple[int, int, int]:
        parts = clause_id.split(":")
        return int(parts[0]), int(parts[1]), int(parts[2])
```

**条款 ID 稳定性策略：**

| 场景 | 策略 |
|------|------|
| **新增条款** | 新 ID，旧条款 ID 不变 |
| **删除条款** | 标记 `deleted=true`，保留 ID 用于追溯 |
| **修改条款** | ID 不变，`content_hash` 变化 |
| **条款拆分** | 原 ID 标记为 `split_from`，新条款生成子 ID（如 `:4:1a`, `:4:1b`） |
| **条款合并** | 原 ID 标记为 `merged_into`，新条款用新 ID |

### 3.3 建议接受/拒绝后的合同状态转换

```
建议 1（必须修改）→ 起草人接受
  → 原文中标记为 <ins>新文本</ins>，旧文本 <del>划线</del>
  → 建议状态：accepted

建议 2（建议修改）→ 起草人修改
  → 起草人在建议基础上自行编辑
  → 建议状态：modified
  → 记录 original_suggestion vs actual_change

建议 3（可选）→ 起草人拒绝（附原因）
  → 建议状态：rejected
  → 原因：暂不修改，因XXX
  → 评审人可查看拒绝原因（但不能覆盖）
```

class RevisionRecord(BaseModel):

    id: str                          # 修订记录 ID

    # 逐建议处理结果
    suggestion_results: List[SuggestionResult]
    # 自行修改部分
    manual_changes: List[ManualChange]
    
    # 状态
    status: str                      # draft / submitted
    submitted_at: Optional[datetime]
    
    class SuggestionResult(BaseModel):

        action: str                  # accepted / modified / rejected
        original_text: str           # 原文
        suggested_text: str          # 建议修改后文本
        actual_text: str             # 实际修改后文本
        reject_reason: Optional[str] # 拒绝原因
        clause_id: str               # 关联条款 ID
    
    class ManualChange(BaseModel):
        """起草人自行修改（非建议驱动）"""
        clause_id: str
        before: str
        after: str
        reason: str

## 四、条款 ↔ 审查条目映射机制

### 4.1 映射模型

```
合同全文
  │
  ├── 第四条 违约责任
  │     ├── 第1款：乙方违约需赔偿甲方一切损失...
  │     │     ├── AI 审查发现 1: 未设置赔偿上限（高风险）
  │     │     │     ↓ 法务确认
  │     │     │     └── 修改建议: 增加上限 ≤30%
  │     │     │           ↓ 起草人接受
  │     │     │           └── v2 修改为: 赔偿直接损失，总额 ≤30%
  │     │     │
  │     │     └── AI 审查发现 2: 包含间接损失（中风险）
  │     │           ↓ 法务忽略（认为合理）
  │     │           └── 状态: ignored（记录忽略原因）
  │     │
  │     └── 第2款：...
  │
  ├── 第六条 管辖约定
  │     └── 第1款：乙方所在地人民法院
  │           ├── AI 审查发现: 异地管辖（中风险）
  │           └── 法务确认 → 建议修改为甲方所在地
  │
  └── ...
```

### 4.2 多对多映射表设计

```sql
CREATE TABLE clause_review_mappings (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    contract_id BIGINT NOT NULL,
    version_id BIGINT NOT NULL,
    clause_id VARCHAR(100) NOT NULL,
    
    -- 条款原文锚定
    clause_chapter VARCHAR(20),       -- 章节号（如"第四条"）
    clause_order INT,                 -- 条款序号
    para_index INT,                   -- 款序号
    content_hash VARCHAR(64),         -- 条款原文哈希
    
    -- 关联的审查发现
    ai_finding_ids JSON,              -- AI 审查发现 ID 列表
    human_opinion_ids JSON,           -- 人工评审意见 ID 列表
    suggestion_ids JSON,              -- 修改建议 ID 列表
    
    -- 状态追踪
    risk_status VARCHAR(20) DEFAULT 'unreviewed',  -- unreviewed / confirmed / resolved / ignored
    last_modified_at TIMESTAMP,
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    UNIQUE KEY uk_clause_version (clause_id, version_id),
    INDEX idx_contract_version (contract_id, version_id),
    INDEX idx_risk_status (risk_status)
);
```

## 五、易用性评估与改进

### 5.1 当前设计评估

| 可用性原则 | 当前状态 | 评分 |
|-----------|---------|------|
| **系统状态可见性** | 流程状态可见，但条款级状态不可见 | 6/10 |
| **系统与现实匹配** | 评审术语专业，但修改建议与原文脱节 | 5/10 |
| **用户控制与自由** | 无 Undo 机制，无分步提交 | 4/10 |
| **一致性与标准** | 术语统一，但评审意见格式不统一 | 7/10 |
| **错误预防** | 无提交前检查，无反悔确认 | 3/10 |
| **识别而非回忆** | 条款列表需记忆编号，无法从原文直达 | 4/10 |
| **灵活性与效率** | 无批量操作，无快捷键 | 3/10 |
| **极简设计** | 评审意见模板冗长，核心信息淹没 | 5/10 |
| **帮助与文档** | 无上下文帮助，无操作提示 | 2/10 |
| **总体** | | **4.3/10** |

### 5.2 重点改进项

| 优先级 | 改进项 | 说明 |
|--------|--------|------|
| **P0** | 三区联动工作台 | 评审人一屏完成：读原文 + 看 AI 标记 + 写意见 |
| **P0** | 条款 ↔ 审查映射 | clause_id 唯一标识，跨版本稳定 |
| **P0** | 行内批注模式 | 评审人在原文上划选文字 → 弹出批注工具栏 |
| **P1** | 修订建议驱动修改 | 起草人逐条 Accept/Reject，而非全文重写 |
| **P1** | 版本对比视图 | v1 ↔ v2 并排显示，变更高亮 + 审查意见同步 |
| **P2** | 快捷键支持 | Tab 切换区域、Ctrl+Enter 提交、ESC 关闭面板 |
| **P2** | Undo 支持 | 评审意见可撤回（限 5 分钟内）、修改可回退 |
| **P2** | 草稿自动保存 | 每 30 秒自动保存评审意见草稿 |

## 六、交互功能详解

### 6.1 评审人核心操作

| 操作 | 触发方式 | 系统响应 |
|------|---------|---------|
| **查看合同** | 点击"开始评审" | 加载三区工作台，左侧原文、右侧 AI 标记列表 |
| **定位条款** | 点击左侧某条款 | 右侧自动滚动到该条款审查面板，条款高亮 |
| **定位原文** | 点击右侧某 AI 标记 | 左侧原文滚动到对应位置，条款黄色高亮 |
| **确认 AI 标记** | 点击面板中"确认"按钮 | 状态从"AI 标记"→"法务已确认"，颜色红→绿 |
| **修正 AI 标记** | 点击"修正" → 编辑 | 修改风险等级 / 描述 / 建议 → 保存 → 标记"人工修正" |
| **新增标记** | 划选原文 → "标记风险" | 弹出新建面板，填写风险描述 → 保存 → 左右侧同步显示 |
| **添加批注** | 划选原文 → "添加评论" | 弹出评论框，支持 @某人、引用法规 → 保存 |
| **提交整体意见** | 处理完所有标记后 | 底部按钮变为可用，选择结论 → 提交 |
| **撤销提交** | 提交后 5 分钟内 | 按钮变为"撤回"，点击撤回 → 状态回退 |

### 6.2 起草人核心操作

| 操作 | 触发方式 | 系统响应 |
|------|---------|---------|
| **查看评审意见** | 打开"我的合同"→ 点击"已退回" | 进入修订工作台 |
| **逐条接受建议** | 点击"接受" | 原文自动替换，`<del>`旧文 + `<ins>`新文 |
| **逐条拒绝建议** | 点击"拒绝" → 填写原因 | 原文不变，建议标记"已拒绝（原因：XXX）" |
| **自行修改** | 直接在原文编辑 | 原文标记为"手动编辑"，区别于建议驱动修改 |
| **批量接受** | 勾选多条 → "批量接受" | 所有勾选建议一次性应用 |
| **保存草稿** | 点击"保存草稿" | 保存当前修改状态，可下次继续 |
| **重新提交** | 点击"重新提交评审" | 生成新版本 → 自动触发 AI 初筛 → 分配法务 |

### 6.3 版本对比操作

| 操作 | 视图 |
|------|------|
| **并排对比** | v1（左）‖ v2（右），变更处高亮 |
| **审查意见叠加** | 右侧 v2 上标注：哪些变更对应哪条评审意见 |
| **逐条导航** | 上一处变更 / 下一处变更 / 仅显示高风险变更 |
| **导出** | 生成 PDF 版本对比报告（用于线下讨论） |

    from_version INT NOT NULL,
    to_version INT NOT NULL,
    created_by BIGINT NOT NULL,
    status VARCHAR(20) DEFAULT 'draft',     -- draft / submitted
    
    suggestion_results JSON,                 -- 逐建议处理结果
    manual_changes JSON,                     -- 自行修改部分
    summary TEXT,                            -- 修订摘要
    
    submitted_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_contract (contract_id),
    INDEX idx_review (review_id)
);
