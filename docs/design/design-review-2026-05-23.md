# 合同审批管理平台 — 设计评审报告

> 日期：2026-05-23 | 评审人：Sisyphus | 版本：V1.0

---

## 一、项目概览

**合同审批管理平台** 是一个全生命周期数字化合同审批管理系统，覆盖"起草→审批→签署→履约→归档"全流程，以 AI 智能审查为核心能力。

### 技术栈
- **前端**：Vue3 + TypeScript（当前以高保真原型 `prototype/` 为准，20 个页面）
- **后端**：FastAPI + Async SQLAlchemy（11 个路由模块）
- **数据库**：MySQL 8（13 核心表）
- **基础设施**：Redis 7、MinIO、Celery、Qwen3.6 本地 AI、Chroma RAG
- **部署**：Docker Compose（Nginx 反向代理）

## 三、数据模型评审

### 3.1 核心表结构（13 表）

| 表名 | 用途 | 关键字段 | 评价 |
|------|------|---------|------|
| `contracts` | 合同主表 | contract_no, status, amount, risk_level | ✅ 字段完整 |
| `contract_versions` | 版本管理 | contract_id, version, content, file_hash | ✅ 支持版本追溯 |
| `approval_flows` | 审批流程 | contract_id, flow_type, current_step | ⚠️ 流程类型硬编码 |
| `approval_steps` | 审批节点 | flow_id, step_number, action | ✅ 节点级追踪 |
| `ai_reviews` | AI 审查 | review_id, clause_reviews(JSON) | ✅ 结构化存储 |
| `risk_alerts` | 风险预警 | alert_type, alert_level, source | ✅ 多维度来源 |
| `audit_logs` | 审计日志 | user_id, action, detail(JSON) | ✅ 全链路记录 |
| `users` / `roles` / `departments` | 权限体系 | username, role_id, department_id | ⚠️ 未实现登录/鉴权 |
| `seal_records` | 用印记录 | contract_id, seal_type, seal_method | ✅ 支持电子/物理 |
| `contract_ledger` | 台账汇总 | 冗余字段便于查询 | ✅ 读写分离思路 |

### 3.2 数据模型问题

1. **外键约束缺失**：`contract_versions.contract_id` 使用 `ForeignKey` 但实际未启用级联删除
2. **JSON 字段过多**：`ai_reviews.clause_reviews`、`risk_alerts.source_detail`、`audit_logs.detail` 均使用 Text 存储 JSON，缺少校验
3. **状态枚举未约束**：`contracts.status` 使用 String(20)，无 CHECK 约束，可能写入非法状态
4. **缺少索引**：`approval_flows` 表缺少 `contract_id` 索引，查询效率低

### 4.2 API 设计问题

1. **鉴权缺失**：所有接口无 `Depends(JWT)` 校验，`creator_id=1` 硬编码
2. **错误处理不统一**：部分接口返回 `{"code": 200, "data": ...}`，部分直接返回 ORM 对象
3. **AI 审查核心未实现**：`_execute_review()` 为空函数，TextExtractor/ClauseParser/RiskScorer 未集成
4. **用印逻辑简化**：`apply_seal()` 仅检查 `approval_status`，未真正创建 seal_record 记录
5. **统计接口硬编码**：`approval_efficiency.bottleneck_nodes` 返回假数据

## 五、业务流程评审

### 5.1 标准业务流程（7 步）

```
① 起草 → ② 提交 → ③ AI 初筛 → ④ 法务评审 → ⑤ 财务评审 → ⑥ 高管审批 → ⑦ 用印 → 归档
```

**条件分支**（基于金额/类型）：
- 采购合同 ≤10 万：简易流程（部门主管→法务→用印）
- 采购合同 10-100 万：标准流程（部门主管→法务→财务→高管→用印）
- 采购合同 >100 万：特殊流程（+董事会审批）
- 销售合同 ≤50 万：简易流程
- 销售合同 >50 万：标准流程
- 劳务合同：不限金额，标准流程

### 5.2 流程问题

1. **AI 审查门禁缺失**：PRD 要求"高风险直接拦截"，但代码中 AI 审查结果不影响审批流程启动
2. **退回机制不完整**：`return` 操作仅设置 `status=returned`，未将合同状态回退到 `draft`
3. **超时升级未实现**：PRD 要求"24h 提醒 + 48h 升级"，代码中无 SLA 计时器
4. **条件分支硬编码**：`total_steps=5` 写死，未根据合同类型/金额动态选择流程模板

## 七、AI 审查模块评审

### 7.1 设计架构

```
合同文件 → TextExtractor(文本提取) → ClauseParser(条款切分)
    → AIReviewEngine(Qwen3.6 LLM) → RiskScorer(风险评分)
    → 写入 ai_reviews 表 + risk_alerts 表
```

### 7.2 审查维度（5 维）

| 维度 | 权重 | 检查项 | 实现状态 |
|------|------|--------|---------|
| 合规性 | 30% | 主体资格、经营范围、强制性规定 | ⚠️ 未实现 |
| 风险条款 | 25% | 违约责任、管辖、解除、保密 | ⚠️ 未实现 |
| 财务条款 | 20% | 付款条件、价格、税务 | ⚠️ 未实现 |
| 履约能力 | 15% | 交付周期、质量标准、验收条件 | ⚠️ 未实现 |
| 异常检测 | 10% | 矛盾条款、隐藏风险、缺失条款 | ⚠️ 未实现 |

### 7.3 种子数据

- 法规知识库：50+ 核心法条（Chroma RAG）
- 风险标签：预定义风险分类
- CUAD 标签桥接：与通用合同分析数据集对齐
- 审查清单：按合同类型预定义检查项
- 修订路由：风险等级→处理策略映射

**评价**：种子数据设计完善，但 AI 引擎核心逻辑（`_execute_review`）为空，V1 阶段需优先实现。

2. **审批流程僵化**：V1 硬编码 3 种流程覆盖 80% 场景，V3 提供可视化设计器
3. **前后端未联调**：原型为纯静态 HTML，后端 API 无鉴权，联调工作量被低估

## 十、改进建议

### 优先级 P0（阻塞上线）

1. **实现 JWT 鉴权中间件**：所有 API 端点需验证 Token，`creator_id` 从 JWT 提取
2. **实现 AI 审查核心逻辑**：`_execute_review()` 需集成 TextExtractor → ClauseParser → AIReviewEngine → RiskScorer 完整流水线
3. **实现文件上传**：MinIO 集成，支持 PDF/Word 上传，计算文件哈希

### 优先级 P1（影响体验）

4. **实现登录/注册**：`system.py` 中 login 接口需对接用户表
5. **完善用印逻辑**：`apply_seal()` 需真正创建 seal_record 并更新合同状态
6. **实现 SLA 超时机制**：审批节点 24h/48h 超时提醒和升级

### 优先级 P2（优化改进）

7. **动态流程模板**：根据合同类型/金额自动选择审批流程，而非硬编码 `total_steps=5`
8. **统一 API 响应格式**：所有接口统一返回 `{"code": 200, "data": ...}` 或 FastAPI 标准 Response
9. **数据库约束**：为 status 字段添加 CHECK 约束，防止非法状态写入
10. **前后端联调**：原型页面对接后端 API，实现真实数据流转
