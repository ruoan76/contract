## 二、架构评审

### 2.1 技术栈选型

```
前端: Vue3 (规划中) → 当前依赖 prototype/ (HTML/JS)
后端: FastAPI + Async SQLAlchemy
数据库: MySQL 8 + Redis 7
存储: MinIO (文件) + Chroma (向量)
任务: Celery (异步)
AI: Qwen3.6 35B-A3B-4bit (本地 MLX)
```

**评价**: 技术栈选型合理。FastAPI 异步适合 I/O 密集型场景，MySQL 8 支持 JSON 字段适合存储审查结果。本地部署 Qwen3.6 符合数据隐私要求。

**风险**: 
- Vue3 前端尚未启动，原型到正式前端的迁移工作量未知
- Qwen3.6 35B 在 Apple Silicon 上推理速度可能成为瓶颈（建议评估量化方案）
- Celery 需要 Redis/RabbitMQ 作为 broker，当前配置中未体现

### 2.2 分层架构

```
┌─────────────────────────────────────────┐
│  前端层: Vue3 / 原型 (20 pages)          │
├─────────────────────────────────────────┤
│  API Gateway: Nginx + CORS + JWT        │
├─────────────────────────────────────────┤
│  后端服务: FastAPI (11 routers)          │
│  ├── contracts / approvals / ai_review   │
│  ├── risks / seals / archives            │
│  ├── audit / statistics / system         │
│  └── ai_review_seeds / ai_review_demo    │
├─────────────────────────────────────────┤
│  基础设施: MySQL / Redis / MinIO / Celery│
└─────────────────────────────────────────┘
```

**评价**: 分层清晰，职责明确。11 个 router 对应 11 个业务域，符合单一职责原则。

**问题**:
1. `ai_review_seeds` 和 `ai_review_demo` 作为独立 router 暴露，生产环境应移除或加权限控制
2. 缺少统一的错误处理中间件（全局异常处理器）
3. 缺少请求限流中间件

**问题**:

2. `contract_ledger` 作为汇总表，未明确是物理表还是视图

4. 缺少 `counterparties` 表模型（DESIGN_STATUS D2 决策独立表，但 models/ 中未见）

### 3.2 状态设计

```
contracts.status: draft → pending → reviewing → approved → signed → archived
                              ↘ rejected / returned → revision → resubmit

contracts.approval_status: pending → approving → approved / rejected / returned
```

**评价**: 双状态设计（主状态 + 审批子状态）合理，支持独立追踪。

**问题**:
1. 缺少 `terminated`（已终止）状态——合同可能中途终止
2. `reviewing` 状态在 D1 决策中被否定（"无 reviewing"），但代码中 `ai_reviews.review_status` 使用 `ai_done | reviewed | confirmed`，与 `contracts.status` 的衔接不清晰
3. 状态转换缺少显式状态机定义（建议用 `transitions` 库或枚举约束）

### 4.2 关键问题

1. **审批节点硬编码**: `_get_next_node` 使用字典硬编码流转顺序，无法支持动态配置
   ```python
   node_sequence = {
       "dept_approval": "legal_review",
       "legal_review": "finance_review",
       # ... 固定死
   }
   ```
   **建议**: 从 `config` 表读取流程模板，支持动态节点配置

2. **AI 审查异步逻辑为空**:
   ```python
   async def _execute_review(contract_id: int, review_id: str):
       # TODO: 实现异步审查逻辑
       pass
   ```
   **影响**: 整个 AI 审查功能无法运行

3. **缺少批量操作**: 合同列表无批量审批、批量导出接口

4. **缺少 WebSocket 实时通知**: 设计文档提到 WebSocket，但 API 路由中未见

5. **文件上传接口缺失**: `contracts` router 无 upload 端点（README 提到 `POST /api/v1/contracts/{id}/upload`）

## 五、业务流程评审

### 5.1 三种流程

| 流程 | 路径 | 适用场景 |
|------|------|----------|
| 简易 | 起草→提交→AI初筛→部门主管→法务快审→用印→归档 | 采购≤10万 |
| 标准 | 起草→提交→AI初筛→法务→财务→高管→用印→归档 | 采购10-100万 |
| 特殊 | 标准流程 + 董事会节点 | 采购>100万 |

**评价**: 流程设计合理，金额阈值可配置。

**问题**:
1. 退回后的再提交路径：是否重跑 AI？设计文档说"可配置"，但无配置入口
2. 并行审批（多部门同时审核）在 PRD 中提到，但流程中未体现
3. 会签机制（D5 决策 V1 不做）的缺失不影响 MVP

### 5.2 退回修订闭环

```
review-workspace / approvals → 退回 → revision-workspace → 修订 → 重新提交
```

**评价**: 闭环清晰，`revision-workspace` 作为下钻页设计合理。

## 六、AI 审查系统评审

### 6.1 架构评价

AI 审查系统是整个项目最复杂也最创新的部分：

| 组件 | 设计 | 实现状态 |
|------|------|----------|
| 文本提取 | PDF/Word/OCR | ⚠️ 代码框架有，未验证 |
| 条款解析 | 正则分段 + 分类 | ⚠️ 代码框架有 |
| 门禁层 | 5 类门禁 (validity/subject/clause/consistency/output) | ❌ 未实现 |
| 五维度 Skill | 合规/风险/财务/履约/异常 | ❌ 未实现 |
| Session 七步 | S0-S7 状态机 | ❌ 未实现 |
| RAG | Chroma + 法规库 | ❌ 未实现 |
| 规则引擎 | 正则 + 阈值 | ⚠️ 代码框架有 |
| 种子数据 | risk_labels/revision_routing 等 | ✅ 已落地 |

### 6.2 关键风险

1. **复杂度极高**: 门禁 + 五维度 + 七步 Session + 三标签体系，V1 8 周内完成 AI 部分极其紧张
2. **本地模型性能**: Qwen3.6 35B 在 Apple Silicon 上逐条款审查可能需数分钟/合同
3. **幻觉控制**: RAG + 规则引擎双保险设计合理，但 Chroma 向量库的召回质量待验证
4. **误报/漏报**: 原型已支持法务标记，但模型自我反思 (Self-Correction) 未实现

### 6.3 建议

- **Sprint 4 优先实现**: 文本提取 + 规则引擎 + 基础报告（先跑通端到端）
- **Sprint AI-1**: 门禁层 + 种子数据导入
- **Sprint AI-2**: RAG + Session 状态机
- 考虑先用云端 API 验证效果，再迁移本地
