# 合同审批管理平台 — 设计与原型对齐要求书

> ⚠️ 页面清单以 [DESIGN_STATUS.md](./DESIGN_STATUS.md) 为准（**20 页**：侧栏 **17** + 下钻 **3**）。`page-risk`、`page-statistics` V1 不建独立页。  
> 版本：V1.2 | 日期：2026-05-18 | 适用范围：产品设计、原型设计、视觉设计、交互设计、前端开发、测试  
> 权限：正式环境见 [permission-matrix.md](./permission-matrix.md)；原型全菜单展示见 DESIGN_STATUS §3.3。

## 一、对齐核心目标

| 目标 | 定义 | 验收标准 |
|------|------|---------|
| **统一性** | 设计内容（视觉、交互）与原型设计的功能、逻辑、布局、文案、交互逻辑完全匹配 | 设计稿与原型逐项核对，偏差率 ≤ 2% |
| **可落地性** | 对齐后的设计贴合原型核心需求，兼顾视觉美观与开发可行性 | 设计稿通过前端开发评审，无"无法实现"反馈 |
| **高效性** | 明确对齐流程和标准，减少协作沟通成本 | 对齐周期 ≤ 3 个工作日，返工次数 ≤ 1 次 |
| **可追溯性** | 所有对齐过程、修改记录留存 | 变更通过 PR 管理，每次修改有 Issue 关联 |

## 二、对齐范围及具体要求

### （一）页面布局对齐

#### 1. 页面整体结构

| 要求 | 说明 | 检查方式 |
|------|------|---------|
| 模块划分 | 设计页面的模块划分需与原型完全一致 | 对照原型 `prototype/index.html` 中的 `page-*` 结构 |
| 模块顺序 | 侧边栏导航顺序、页面内容区模块顺序不得调整 | 核对 20 个页面的模块排列 |
| 模块间距 | 模块间 padding/margin 与原型保持一致（原型使用 12px/16px/20px/24px 四档） | 设计稿标注间距值，开发核对 CSS |

**侧栏导航（17 项）** — 顺序与 `prototype/src/shell.html` 一致：
```
dashboard · messages · create · contracts · templates
ai-review · clause-compare · review-center · review-workspace · review-history
approvals · seal · archives · counterparties · config · users · audit
```

**下钻页（3 项，无侧栏菜单）**：
```
page-contract-detail（合同详情）
page-approval-history（单合同审批历史时间线）
page-revision-workspace（修订工作台）
```

**V1 不建独立页（功能合并）**：
- 风险预警 → `messages` + `contract-detail` + `ai-review`
- 统计看板 → `dashboard` + `review-center`

#### 2. 元素布局

| 要求 | 说明 | 检查方式 |
|------|------|---------|
| 按钮位置 | 所有 293 个按钮的位置、大小、排列方式与原型一致 | 设计稿标注按钮坐标，开发核对 DOM 结构 |
| 输入框 | 表单字段顺序、输入框宽度、placeholder 文案与原型一致 | 对照 `page-contract-create` 等表单页面 |
| 图标 | 图标样式、尺寸、位置与原型一致（原型使用 emoji 图标，设计需替换为 SVG） | 设计稿提供 SVG 图标清单，开发核对 |
| 表格 | 列顺序、列宽比例、操作列位置与原型一致 | 对照各列表页面 |

#### 3. 响应式布局

| 要求 | 说明 |
|------|------|
| PC 端 | 主设计稿，侧边栏 240px + 内容区自适应 |
| 移动端 | V2 规划，当前原型无移动端页面 |
| 适配规则 | 若 V2 增加移动端，需对齐各终端的布局适配逻辑 |

#### 3. 异常场景

| 异常类型 | 原型处理 | 设计需对齐 |
|---------|---------|-----------|
| 表单校验失败 | showToast('⚠️ 请填写必填项') | 设计稿标注错误提示样式 |
| 无数据 | 空状态提示（设计稿需提供） | 设计稿提供空状态插画/文案 |
| 加载失败 | 重试按钮（设计稿需提供） | 设计稿标注加载状态 |
| 超时 | SLA 倒计时 + 超时变红 + 闪烁动画 | 设计稿标注超时状态视觉 |
| 权限不足 | **正式环境**：侧栏按 RBAC 隐藏菜单；**原型**：17 项全展示，无权限项 🔒（仍可点进走查） | 设计稿标注 [permission-matrix.md](./permission-matrix.md) 规则；原型不画「菜单消失」态 |

## 四、执行流程

### 阶段一：设计前对齐（1 天）

```
1. 原型设计师提供最新原型 + 标注文档
2. 视觉/交互设计师阅读原型，标记疑问
3. 产品/原型/设计三方会议，确认对齐范围
4. 输出《对齐确认清单》（签字确认）
```

### 阶段二：设计中对齐（2-3 天）

```
1. 视觉设计师按页面逐个设计，每完成 3-5 页提交一次评审
2. 原型设计师核对设计稿与原型一致性
3. 发现偏差 → 提交 Issue → 三方确认 → 修改
4. 输出《设计稿评审记录》
```

### 阶段三：设计后对齐（1 天）

```
1. 前端开发核对设计稿与原型一致性
2. 发现偏差 → 提交 Issue → 三方确认 → 修改
3. 测试工程师执行对齐测试
4. 输出《最终对齐报告》
```

### 阶段四：开发中追溯（持续）

```
1. 开发过程中发现设计/原型问题 → 提交 Issue
2. 产品/原型/设计三方评审 Issue
3. 确认修改 → 更新原型 → 更新设计稿 → 开发实现
4. 所有变更通过 PR 管理，关联 Issue
```

## 六、追溯机制

### 变更管理

```
所有变更必须通过以下方式管理：
1. Issue 跟踪：每个变更创建 Issue，描述变更原因、影响范围
2. PR 关联：代码变更通过 PR，关联对应 Issue
3. 设计稿版本：设计稿按版本管理，标注变更内容
4. 原型更新：原型更新需通知所有相关方
```

### 偏差记录

| 偏差类型 | 记录内容 | 责任人 | 处理方式 |
|---------|---------|--------|---------|
| 布局偏差 | 页面、模块、偏差描述、影响 | 视觉设计师 | 修改设计稿 |
| 功能偏差 | 页面、功能、偏差描述、影响 | 前端开发 | 修改代码/更新原型 |
| 文案偏差 | 页面、文案位置、原内容、新内容 | 产品经理 | 确认文案并更新 |
| 交互偏差 | 页面、交互、偏差描述、影响 | 交互设计师 | 修改交互说明 |

### 复盘机制

```
每个里程碑结束后：
1. 汇总本阶段偏差记录
2. 分析偏差原因（设计疏忽/原型不清/沟通不足）
3. 制定改进措施
4. 更新对齐要求书
```
